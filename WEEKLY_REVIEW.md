# 每週檢視 Runbook(給下一個 session 的 Claude / 或人工執行)

> 報告本身每週六 09:00 已由 Actions 自動產生並 commit(`reports/validate_<資料迄日>.md`)。
> 本文件是「讀報告 → 判斷 → 行動」的標準程序。**判斷依據一律用報告的 OOS 欄**,
> IS(2026-07-05 前)數字只做背景參考。

## 0. 前置檢查

1. `git pull`;確認 `daily-fetch` 與 `weekly-validate` 兩個 Actions 最近一次都是綠的。
   紅的先修管線(看 stderr 警告:TAIEX 抓空、price_adj 缺列、無事件跳空),當週不做策略判斷。
2. 打開最新 `reports/validate_*.md`。看第一行 OOS 天數:**< 10 交易日 → 只讀不動**,
   照樣寫週記(見 §3)。

## 1. 讀報告的順序與行動判準

| 節 | 看什麼 | 行動門檻(未達 = 只記錄) | 行動 |
|---|---|---|---|
| ① 元素 IC | composite_s 的「族群內·OOS」 | 連 2 週為負 → 追查;單一元素 OOS IC 連 3 週為負 | 該元素權重下調一檔(例 0.8→0.5);上調同樣要 3 週證據 |
| ② tier | 真弱 OOS 超額應為負;真強/蓄勢應為正 | 連 3 週反向 | 檢討該 tier 條件(先排除 regime 因素再動) |
| ③ 濾網 cohort | 放行 vs 擋下(OOS) | OOS ≥15 筆且「領跌<0」組不再顯著落後 | 放寬蓄勢抗跌條件(down_rs20≥0 → ≥−0.5% 或移除,擇一);注意:39 檔擴充後 in-sample 已反轉,此題優先觀察 |
| ④ 族群層 | med_dip 領漲命中(OOS) | 連 4 週 ≤ 基準(33%) | med_dip 從「選族群主訊號」降級為輔助 |
| ⑤ 市值同尺 | 影子因子 rs20/σ20 vs 原始(OOS) | 連 4 週優於原始且差距 ≥0.02(小型股行情期的證據加倍計) | s_price 改用風險調整值(score.py);tier 市值佔用持續與領漲結構不符 → 開 issue 檢查 |

## 2. 行動規範(鐵律)

- **每次最多動 1~2 個旋鈕**;不同層(權重 vs tier 條件)不同週動。
- 動旋鈕 = 改 `score.py` CONFIG(個股層)或 `fetch_daily.py` 頂部旋鈕(族群/大盤層)
  → 本地重跑 `score.py` + `build_dashboard.py` 確認 → commit(訊息附依據數字)→ 記 `CHANGELOG.md`。
- **改了權重或 tier 條件後,必須把 `validate.py` 的 `IS_CUTOFF` 改成當天**——
  否則之前的 OOS 對新規則而言其實是 in-sample,會被重複當證據。
- 絕不做:憑單週數據調整;用 IS 欄數字當調整依據;同週改權重又改 tier 定義。
- 每季第一個週六:先跑 `scripts/screen.py` 走 README「Universe 治理」流程,再做週檢視。

## 3. 記錄

- **有動作** → CHANGELOG 記一節(改了什麼、依據哪份報告哪個數字)。
- **無動作** → 也要留痕:在當週 commit(或直接在 memory)記一行
  「YYYY-MM-DD 週檢視:OOS n 日,無行動,理由」。連續「無行動」是正常狀態,
  前 2~4 週預期如此。

## 4. 目前懸而未決、等 OOS 裁決的問題(2026-07-05 起算)

1. **蓄勢抗跌濾網存廢**(③):舊 30 檔 in-sample 支持、39 檔擴充後反轉——OOS 定生死。
2. **市值同尺 / 影子因子**(⑤):本窗兩把尺幾乎等價,等小型股行情期的分歧證據。
3. **警告類 tier(強但過熱/陷阱)方向**:主升段中曾為反指標,需經歷完整漲跌循環。
4. **權重再校準**:OOS 累積 ≥4 週(約 2026-08-02 之後)才做第一次認真的權重檢視。
